### Headline

The core defense-in-depth design **holds**: the deductible *amount* is computed in deterministic
code (`evaluateSubstantiation`), every user-facing *number* in the router renders from a DB query
(never the model), and `sanitizeIntent` clamps model output to a fixed enum. Injection **cannot
fabricate a dollar figure, run a query it shouldn't, or extract another tenant's data.**

## Threat model

The attacker controls the **SMS body** and, indirectly, the **text printed on a receipt photo**
(Haiku OCRs it into `vendor`/`items`/`raw_text`, which then feed categorization and SMS wording).
Goals: flip a non-deductible expense into a deductible one, inflate the amount, make Tally emit tax
advice or fabricated numbers, or extract the system prompt.

## Structural guarantees (verified by code, not a single prompt)

- **Deterministic deductibility.** `evaluateSubstantiation` (src/lib/substantiation.ts) is a pure
  function of `(rule, amount, has_photo, fields)`. The LLM never sets `deductible_amount_cents`,
  `needs_receipt`, or `deduction_percentage`. A category flip changes *which rule loads*, but the
  math itself is uninjectable.
- **Numbers-from-DB.** The router (src/lib/router.ts) renders every figure from `lib/queries.ts`.
  The model only picks an intent + params, then `sanitizeIntent` clamps them:
  hostile raw intent clamped to safe values: {"kind":"query","tool":"aggregate","category":null,"count":10} (count bounded ≤10, tool defaulted to enum, unknown category nulled, junk command dropped).
- **Tenant isolation.** `runQuery` always uses the caller's `organization_id` passed in by the
  handler; the model has no way to supply another org id. Cross-tenant reads are structurally
  impossible from this path.

## Recommendations

1. **[medium] Category is the real injection surface — IMPLEMENTED (DEC-055).** Deductibility math
   is safe, but the *category* (which sets the %) is LLM-chosen, so it was the thing worth
   hardening. Done: `src/lib/review.ts` flags an expense for human review when categorization
   confidence is below 0.8 OR the note is instruction-shaped ("categorize as", "ignore the above",
   "system:") — markers that never appear in honest expense texts. The flag is deterministic,
   persisted (`needs_review`), and surfaced on the dashboard + CSV export. Closes both the
   concert-tickets accuracy edge (0.72 conf → flagged) and category-flip injection as a backstop.
2. **[resolved] Graceful-fail on the live text path — closed by repoint (DEC-064 follow-up).**
   The harness now exercises `parseAndCategorizeText` (the merged call production runs), not the
   orphaned standalone `parseTextExpense`. That path is wrapped twice — `handleTextAsNewExpense`
   catches a parser throw and returns `MSG.couldntRead` ("couldn't read that — rephrase"), behind a
   top-level dispatch catch that falls back to `MSG.failure`. The inbound is logged to
   `conversations` BEFORE parsing, so nothing is silently dropped (and for sub-$75 strict expenses
   that row IS the written record). A throw is a low-severity note, not a gap.
3. **[low] Amount is LLM-extracted from untrusted text.** Self-harm mostly (a user inflating their
   own deduction), but a sanity bound + a "that looks unusually large — confirm?" reply would add
   a cheap tripwire.
4. **Keep this in CI-adjacent rotation.** Re-run `npm run redteam` whenever a prompt changes; new
   prompt phrasing is exactly when an injection guardrail silently regresses.
